# 고등부 영수증 관리 시스템 -- 데이터 모델

> 앱에서 다루는 핵심 데이터의 구조를 정의합니다.

---

## 전체 구조

```
[사용자(User)]
      |
      ├── 1:N ──> [장부(Ledger)] ──> 1:N ──> [장부항목(LedgerEntry)]
      |                                            |
      ├── 1:N ──> [영수증(Receipt)] ───────────────┘ (선택적 연결)
      |                 |
      |                 ├── N:1 ──> [분류(Category)]
      |                 └── N:1 ──> [결산기(Settlement)]
      |
      └── role: master / accountant / teacher
```

> **장부와 영수증은 별도 개념:**
> - 장부(Ledger): 회계 장부 단위 (전체회기, 수련회, 전도축제 등)
> - 장부항목(LedgerEntry): 장부의 개별 행. 영수증 승인 시 자동 생성되거나, 회계 교사가 직접 추가/수정/삭제 가능
> - 영수증(Receipt): 사진 기반 제출물. 승인 시 장부항목으로 자동 반영되지만, 장부항목은 영수증 없이도 존재 가능

> 영수증 사진은 Supabase Storage에 저장, `image_url`로 연결.

## 엔티티 상세

### 사용자 (User)
시스템에 접근하는 모든 사람. 계정 신청 후 master 승인을 받아야 사용 가능.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (Supabase Auth UID) | uuid | O |
| email | 아이디 (이메일, 로그인용) | "kim@church.com" | O |
| name | 실명 | "김교사" | O |
| department_id | 소속 부서 | "고등부" | O |
| position | 교회 직분 (신청 시 본인 선택) | "교사" / "부장교사" / "총무교사" / "회계교사" / "담당교역자" | O |
| role | 시스템 역할 (master 승인 시 부여) | "master" / "accountant" / "teacher" | X (승인 전 null) |
| status | 계정 상태 | "pending" / "active" / "inactive" | O |
| created_at | 신청일 (자동) | 2026-03-19 | O |

> **직분(position)과 역할(role)은 다른 개념:**
> - `position`: 교회 내 직책 (담당교역자, 부장교사, 총무교사, 회계교사, 교사 등) — 표시용
> - `role`: 시스템 권한 (master / accountant / teacher) — 기능 접근 제어용
> - 예: 직분이 "담당교역자"라도 시스템 역할이 teacher면 열람+제출만 가능

**계정 상태 흐름:**
```
계정 신청 → [pending] 승인 대기 (시스템 접근 불가)
                ↓
           master 승인 + 시스템 역할 부여
                ↓
            [active] 사용 가능
                ↓ (필요 시)
           [inactive] 접근 차단
```

> `pending` 상태에서는 "승인 대기중" 화면만 표시. 어떤 기능도 사용 불가.
> `role`은 master가 승인할 때 함께 부여 (teacher / accountant).

**계정 신청 입력 항목:**
| 항목 | 설명 |
|------|------|
| 아이디 (이메일) | 로그인용 이메일 |
| 비밀번호 | Supabase Auth 관리 |
| 실명 | 본인 이름 |
| 직분 | 선택: 교사 / 부장교사 / 총무교사 / 회계교사 / 담당교역자 |

> Phase 1에서 소속 부서는 "고등부" 자동 배정 (선택 UI 없음).

**역할 정의:**

| 역할 | 코드 | 설명 |
|------|------|------|
| 마스터 | `master` | 최고 권한. 모든 부서·모든 데이터 CRUD, 계정 승인, 시스템 설정 |
| 회계 교사 | `accountant` | 자기 부서 장부 CRUD, 영수증 승인·반려, PDF, 엑셀 |
| 교사 | `teacher` | 자기 부서 열람(읽기 전용), 영수증 제출, 자기 제출 내역 조회, PDF 열람·다운로드·인쇄 |
| 감사 | `auditor` | **(Phase 3)** 전체 부서 열람만 (수정 불가). 교육위원장(장로) 등 |

### 장부 (Ledger)
회계 장부 단위. 전체회기 장부(기본)와 특수 목적 장부(수련회, 전도축제 등)를 구분.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | ldg-001 | O |
| department_id | 소속 부서 | "고등부" | O |
| name | 장부명 | "전체회기" / "2026 수련회" / "전도축제" | O |
| type | 장부 유형 | "main" / "special" | O |
| description | 설명 | "2026년 여름 수련회 전용 장부" | X |
| is_active | 활성 여부 | true / false | O |
| created_by | 생성자 User ID | uuid | O |
| created_at | 생성일 (자동) | 2026-03-19 | O |

> **장부 유형:**
> - `main`: 전체회기 장부 (부서당 1개, 삭제 불가)
> - `special`: 특수 목적 장부 (수련회, 전도축제 등, 자유 생성/종료)

---

### 장부항목 (LedgerEntry)
장부의 개별 행. 영수증 승인 시 자동 생성되거나, 회계 교사가 엑셀처럼 직접 추가/수정/삭제 가능.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | ent-001 | O |
| ledger_id | 소속 장부 | ldg-001 | O |
| receipt_id | 연결된 영수증 (없을 수 있음) | rct-001 / null | X |
| category_id | 분류 | cat-001 | O |
| date | 거래 날짜 | 2026-03-15 | O |
| description | 항목 내용 | "3월 셀 간식비" | O |
| income | 수입액 | 0 | O |
| expense | 지출액 | 48000 | O |
| memo | 비고 | "카드 결제" | X |
| source | 항목 출처 | "receipt" / "manual" / "excel_import" | O |
| created_by | 작성자 User ID | uuid | O |
| created_at | 작성일 (자동) | 2026-03-19 | O |
| updated_at | 수정일 (자동) | 2026-03-19 | O |

> **항목 출처(source):**
> - `receipt`: 영수증 승인으로 자동 생성된 항목
> - `manual`: 회계 교사/master가 직접 입력한 항목
> - `excel_import`: 엑셀 가져오기로 생성된 항목
>
> **잔액(balance)**: DB에 저장하지 않음. 조회 시 날짜순 누적 계산 (수입 - 지출).
> **직접 편집**: 회계 교사 이상이 행 추가/수정/삭제 가능. 영수증 연결 항목도 금액·내용 수정 가능.
> **영수증↔장부항목 독립성**: 영수증 수정/삭제 시 이미 생성된 장부항목은 영향 없음 (독립 관리).
> **결산 기준**: 결산·PDF는 LedgerEntry 기준으로 집계 (Receipt 기준 아님).
> **PDF 출력**: 지출 항목만 출력. 수입 항목은 PDF에 포함하지 않음. 영수증 없는 항목은 이미지 자리에 "[영수증 없음]" 표시.
> **특수 장부 편의 기능**: 특수 장부 종료 시 전체 수입·지출 합산을 전체회기 장부에 1건으로 반영하는 기능 검토 (예: "2026 수련회 정산 ₩350,000")

### 결산기 (Settlement)
담당 교역자가 직접 기간을 설정하는 결산 단위.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | smt-2026-01 | O |
| title | 결산 제목 | "2026년 전반기 결산" | O |
| start_date | 결산 시작일 | 2026-01-01 | O |
| end_date | 결산 종료일 | 2026-05-31 | O |
| memo | 비고 | "수련회 포함" | X |
| created_by | 생성한 사용자 ID | uuid (master) | O |
| created_at | 생성일 (자동) | 2026-01-01 | O |

> 결산 기간은 고정이 아니라 **담당 교역자가 직접 설정**합니다.
> 전반기/후반기 같은 이름도 자유롭게 지정 가능.

### 영수증 (Receipt)
지출 1건. 교사가 제출하거나 회계 교사/교역자가 직접 입력.
**승인(approved) 상태만 장부·결산에 반영.**

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | rct-0001 | O |
| department_id | 소속 부서 (타 부서 확장 대비) | "고등부" | O |
| settlement_id | 소속 결산기 | smt-2026-01 | X (승인 시 배정) |
| category_id | 분류 | cat-001 | O |
| submitted_by | 제출자 User ID | uuid | O |
| status | 처리 상태 | "pending" / "approved" / "rejected" | O |
| date | 지출 날짜 | 2026-03-15 | O |
| subtotal | 소계 (할인·배달비 전 금액) | 50000 | X |
| discount | 할인금액 | 5000 | X |
| delivery_fee | 배달비 | 3000 | X |
| final_amount | **최종 지불 금액 (결산 기준값)** | 48000 | O |
| vendor | 가게·업체명 | "이마트" | X |
| description | 지출 내용 | "3월 셀 간식비" | O |
| image_url | 영수증 사진 URL (Supabase Storage) | https://…/rct-0001.jpg | X |
| ocr_raw | OCR 원본 추출 텍스트 | "2026.03.15 ₩48,000" | X |
| reviewed_by | 검토한 회계 교사/교역자 User ID | uuid | X |
| reviewed_at | 승인/반려 처리 일시 | 2026-03-18 14:30 | X |
| reject_reason | 반려 사유 | "금액 불일치" | X |
| memo | 비고 | "카드 결제" | X |
| pdf_crop | PDF 이미지 배치 조정값 (jsonb) | { "scale": 1.5, "offsetX": 20, "offsetY": 40 } | X |
| created_at | 제출일 (자동) | 2026-03-18 | O |
| updated_at | 수정일 (자동) | 2026-03-18 | O |

> **금액 계산**: `final_amount = subtotal - discount + delivery_fee` (직접 입력도 가능)

## 상태(status) 흐름도

```
교사/회계교사가 제출
        ↓
    [pending]  ← 미승인, 장부 미반영
        ↓                ↓
   [approved]        [rejected]
   → LedgerEntry 자동 생성   반려 사유 기록
     (source: "receipt")     제출자에게 표시
   결산에 포함

* master는 모든 상태의 영수증을 수정·삭제 가능
* 회계 교사는 LedgerEntry를 엑셀처럼 직접 추가·수정·삭제 가능 (source: "manual")
```

## 권한 매트릭스 (데이터 접근)

| 엔티티 | master | accountant | teacher |
|--------|:------:|:----------:|:-------:|
| User | **CRUD (전체)** | 조회 (본인) | 조회 (본인) |
| Ledger | **CRUD (전체)** | 생성·종료 + 조회 (자기 부서) | 조회 (자기 부서, 읽기 전용) |
| LedgerEntry | **CRUD (전체)** | CRUD (자기 부서) | 조회 (자기 부서, 읽기 전용) |
| Settlement | **CRUD (전체)** | 조회 + PDF 생성 | 조회 + PDF 열람·다운로드·인쇄 |
| Receipt | **CRUD (전체)** | 승인·반려 + 직접 입력 (자기 부서) | 제출 + 자기 것만 조회 |
| Category | **CRUD** | 조회 | 조회 |
| Position | **CRUD** | 조회 | 조회 |
| ExcelSync | **CRUD** | 실행 + 조회 (자기 부서) | ❌ |

## 설계 근거

- **User 엔티티**: 역할 기반 권한 관리, Supabase Auth 연동으로 인증 처리
- **role 필드**: 단순 문자열 3종류로 충분 (복잡한 RBAC 불필요)
- **submitted_by / reviewed_by**: User ID 참조로 누가 제출·검토했는지 추적
- **결산기 자유 설정**: year/half 대신 start_date/end_date + title로 유연하게 대응
- **장부 분리**: 전체회기(main) + 특수 목적(special)으로 이벤트별 독립 장부 관리
- **장부항목 직접 편집**: 영수증 연동뿐 아니라 엑셀처럼 행 단위 CRUD 가능
- **department_id 선확보**: Phase 3 타 부서 확장 대비

## 엑셀 장부 컬럼 매핑

기존 엑셀 장부 구조 확인 완료. 가져오기/내보내기 시 아래 매핑 적용.

**수입/지출 구분**: 엑셀에서는 수입과 지출이 한 장부에 함께 존재.
시스템에서는 LedgerEntry로 통합 관리하되, 카테고리로 수입/지출을 구분.

| 엑셀 컬럼 | LedgerEntry 필드 | 비고 |
|----------|-----------------|------|
| 항목 | `description` | 거래 내용 |
| 수입액 | `income` | 수입 금액 |
| 지출액 | `expense` | 지출 금액 |
| 잔액 | 자동 계산 | 누적 합계 (DB 저장 안 함, 조회 시 계산) |
| 카테고리 | `category_id` | 아래 분류 매핑 참고 |

**분류(Category) 확정 — 수입·지출 통합:**

| 순서 | 분류명 | 유형 | 사용 예 |
